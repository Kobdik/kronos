## Сервис синхронизации на основе Apache Kafka
Имитационное моделирование управления материальными ресурсами использует сервис синхронизации, 
который уведомляет подписчиков о наступлении нового дня, после чего, 
поставщики материалов отправляют запланированные на этот день поставки,
а потребители материалов - заявки. Склады следят за датами обращений, а также отгружают материалы, 
если на текущий момент в наличии достаточно остатков запрашиваемых материалов.

Подробнее тут:
[Имитационное моделирование](https://github.com/Kobdik/delivery)

Сервис синхронизации реализован как продьюсер Apache Kafka, пишущий в топик, на который подписаны другие сервисы-потребители.
Проблема в том, что на моем ноутбуке, где развернут кластер из 3-брокеров Kafka, в лучшем случае, события приходят подписантам 
с разбросом во времени от 10 до 30 миллисекунд. Если я хочу обработать до сотни событий имитируемого дня за 100 миллисекунд, 
то треть отведенного времени будет потеряна. Из-за рассинхронизации возникают коллизии, при которых поставки или заявки на склад 
приходят раньше или позже текущего дня и не могут быть вовремя обработаны.

Потоки сообщений в Kafka несомненно быстрые, вероятно, что более ста сообщений в миллисекунду. 
Однако, эти потоки в разных партициях идут относительно друг-друга со сдвигом во времени, 
в добавок ко всему, имеется некоторая постоянная задержка прохождения пути от продюсера до консьюмера.


## Сервис синхронизации на gRPC

Использование потоковой передачи с сервера (Server streaming gRPC) решает проблему синхронизации с точностью до 2-х миллисекунд.
Существенно меньше задержка, никаких сдвигов во времени между соединениями. 

Интересна сама задача организации сервиса. 
На стороне сервера реализован генератор "дней":
1. Задаем шаг или тактовую длительность одного события, например, 100 миллисекунд.
2. С интервалом в 100 шагов генерируем серию событий.
3. Каждая серия состоит из пустого шага, одного инициирующего, еще 8-ми пустых и 60-ти последовательных событий. 

```
service Kron {
    // get ready
    rpc Ready(EmptyReq) returns (Cell) {}
    // returns stream of cells
    rpc Band (EmptyReq) returns (stream Cell) {}
}
```

Метод Ready возвращает клиенту в одном из полей время в миллисекундах, когда начнется генерация следующей серии событий, 
а также статус сервиса и номер текущей генерируемой серии. Клиент, зная время следующей серии, вызывает метод Band прямо к началу генерации. 

Каждая серия начинается с пустого такта, чтобы клиент успел установить соединение, затем идет начальное событие, 
которое служит сигналом для инициализации, чтобы поставщики и потребители материалов начали загрузку своих планов поставок 
и заявок на потребности в материалах. Далее, идут 8 пустых тактов (шагов), чтобы загрузка успела завершиться до начала серии из 60 событий, 
обозначающих начала последовательности дней с 2025-08-01 по 2025-09-29 включительно.

При вызове на стороне сервера, функция Band запускается в отдельной горутине, внутри горутины обновляется atomic-счетчик количества 
соединившихся с сервером клиентов, чтобы при генерации событий знать сколько копий текущего события записывать в канал.
С началом генерации каждая горутина с Band читает из общего канала события и отправляет их в свой выходной поток. 

В итоге, получаем систему, в которой клиенты получают события из потоков с высокой степенью синхронности.
```
1770136674826 keys:"wait" val:5 cmd:"calend" dms:1770136674834
request Band after 8 ms
request Ready after 8 ms
call of Band at 1770136674836
read stream..ns: 5, day:0, с2025, 8924 -> 8925
read stream..ns: 5, day:1, 2025-08-01, 9824 -> 9825
read stream..ns: 5, day:2, 2025-08-02, 9924 -> 9924
read stream..ns: 5, day:3, 2025-08-03, 10024 -> 10024
read stream..ns: 5, day:4, 2025-08-04, 10124 -> 10124
read stream..ns: 5, day:5, 2025-08-05, 10224 -> 10225
```
Первая строчка - ответ на запрос Ready, где 1770136674826 - текущий момент времени (ms), 
val:5 - номер и dms:1770136674834 - время начала генерации следующей серии, разница 8 (ms).

request Band after 8 ms - cоответственно, вызов Band был запланирован через 8 (ms).

call of Band at 1770136674836 - время открытия потока в методе Band, заняло 2 (ms), при запасе времени на соединение в 100 (ms).

read stream..ns: 5, day:0, с2025, 8924 -> 8925 - инициирующее событие за 9 тактов до начала 60-ти элементной серии.

read stream..ns: 5, day:1, 2025-08-01, 9824 -> 9825 - события в потоке с интервалом в 100 (ms), 
где ns: 5 - номер серии, 9824 -> 9825 - маркеры времени (ms) в моменты отправки -> приема.
